Two features, three design decisions

Pick an approach for each. I'll recommend, but your call.

① Where do uploaded URLs live in the codebase?

A

One images.ts per app ★ recommended

Each site gets apps/deck-repair/src/data/images.ts exporting heroImages (page → URL map) and galleryImages (array). Pages import from it. Photo-picker writes/updates this file after upload.

  • One place to look per site
  • Easy to bulk-edit manually
  • Consistent pattern across all 11 apps
  • Pages need a one-time refactor to import from it
B

Separate per-category files

hero-images.ts, gallery-images.ts, etc. per app. More granular but more files to manage.

  • Each category fully isolated
  • 11 apps × N categories = many files
  • Pages import from multiple places
C

Keep hardcoded, just log URLs

Don't touch page files at all. Photo-picker just shows a copyable URL log — paste manually where needed.

  • Zero codebase changes
  • Fastest to ship
  • Manual copy-paste per page
  • No bulk hero assignment

② How does photo-picker generate gallery components?

A

Generate ready-to-paste Astro snippet ★ recommended

After uploading 4+ gallery images, a "Generate Gallery" button appears. It writes apps/deck-repair/src/data/images.ts with the galleryImages array, and shows a ready-to-paste <ServicesGallery /> snippet with all props filled in.

B

Generate a standalone SiteGallery.astro

Writes a full Astro component file per site with the images hardcoded inside. Drop <SiteGallery /> anywhere.

③ Component preview — how accurate does it need to be?

A

Inline HTML mock in photo-picker ★ recommended

After selecting category + microsite, the center panel shows the uploaded image rendered inside a faithful mockup of the target component (hero section, gallery card, etc.). No Astro dev server needed.

B

Live iframe to Astro dev server

Embed the running pnpm dev site in an iframe and hot-reload a preview page with the new image. Pixel-perfect but requires dev server running alongside.
